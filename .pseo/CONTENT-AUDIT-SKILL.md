# Content Audit — Blog Health Check Skill

## Overview

This skill runs a **full audit** of all blog content and produces an actionable report. It's the diagnostic layer that feeds into `improve-content` for fixes.

**Cycle:** `niche-research` → `pseo-20` → **`content-audit`** → `improve-content` → repeat

## Prerequisites

- **`keyword-map.json`** must exist in `public/assets/` (generated by `scripts/keyword-map.js` during build)
- **`keyword-metrics-cache.json`** in `.pseo/` — automatically applied at build time if present. Populated via Ahrefs MCP during skill execution.
- **`analytics-snapshot.json`** in `.pseo/` — live traffic + ranking data from PostHog and GSC. Generated by `scripts/analytics-snapshot.js` (Step 0). Config: `.pseo/analytics-config.json`.
- Optional: **Ahrefs MCP** server configured (see `.mcp.json.example`) for live ranking data cross-reference

## When This Skill Triggers

- User says: "audit", "health check", "content review", "check cannibalization", "SEO audit"
- Periodically (recommended: after every batch of new content)
- Before expanding to new verticals (baseline check)
- After applying improvements from `improve-content`

## What This Skill Checks

### 1. Keyword Cannibalization

**Source:** `public/assets/keyword-map.json` (auto-generated at build time)

Read the keyword map and report:
- All conflicts (keywords used by 2+ pages)
- Risk level: HIGH (same industry, same keyword) vs MEDIUM (cross-industry overlap)
- Specific fix recommendations per conflict

**Visual:** Direct user to `/seo-dashboard` for interactive tree view.

### 2. Interlinking Health

**Source:** All content JSON files

For each page, count `<a href` occurrences in body text fields (descriptions, context, body):
- Target: 2-4 contextual inline links per page
- Flag pages with 0-1 links (under-linked)
- Flag pages with 5+ links (over-linked)
- Check that linked URLs actually correspond to existing pages
- Check for broken internal links (pointing to pages that don't exist)

**Scan fields by content type:**

| Type | Fields to scan for `<a href` |
|------|------------------------------|
| resource | `content.intro.context`, `content.sections[].items[].description`, `pro_tips[].tip` |
| guide | `content.intro.problem_statement`, `content.sections[].body`, `content.sections[].actionable_tip` |
| tool | `content.intro.body`, `content.how_to_use[].description`, `content.use_cases[].description` |
| alternatives | `content.intro.why_look_for_alternatives`, `content.alternatives[].key_differentiator` |
| comparison | `content.intro.context`, `content.criteria[].details` |

### 3. External References

**Source:** All content JSON files

For each page:
- Count `external_references` (target: 3-5)
- Check tier distribution (need at least 2 Tier 1 sources with DA 80+)
- Verify no competitor links (defined in `site.config.json`)
- Flag duplicated references across pages (same URL used in 3+ pages = lazy)

### 4. Content Depth

**Source:** All content JSON files

For each page:
- Section count (resources: 3-5 sections, guides: 6-8)
- Items per section (resources: 5-8 items each)
- Description length (3-4 sentences per item description)
- Pro tips count (exactly 5)
- Common mistakes count (exactly 3)
- FAQ count (where applicable: exactly 5-6)
- Industry-specific test: swap the industry name — does the content still make sense? If yes → too generic

### 5. Schema Compliance

**Source:** All content JSON files vs `.pseo/schemas/*.ts`

For each page, validate:
- All required fields present
- Field counts within constraints (min/max items)
- Enum values valid (difficulty: beginner/intermediate/advanced, impact: high/medium/standard)
- SEO description ≤ 155 characters
- 8-12 keywords in seo.keywords
- Title follows deterministic template

### 6. Coverage Analysis

**Source:** `.pseo/taxonomy/_registry.json` + existing content files

- Which verticals have content vs which are planned but missing?
- Which content subtypes exist per vertical?
- Batch rollout progress (Batch 1-4 status)
- Gap identification: "Health vertical has 0 pages, was planned for Batch 2"

### 7. Live Rankings (Optional — requires Ahrefs MCP)

When the Ahrefs MCP is available, use the Site Explorer organic-keywords MCP tool with: `target={your-domain}` (get domain from `site.config.json`), `mode=domain`, `select=keyword,volume,position,traffic,url`, `limit=200`, `country=us`.

**Tip:** Call the `doc` tool first to learn the MCP tool's interface. Monetary values are returned in USD cents — divide by 100 for USD.

Cross-reference with content:
- Pages with 0 organic keywords (not indexed or not ranking)
- Pages ranking 4-10 (opportunity to push to top 3)
- Pages ranking 11-30 (need content improvements)
- Keywords we rank for that aren't in any page's `seo.keywords` (accidental rankings — capture them!)
- Pages with high impressions but low CTR (title/description needs improvement)

## Audit Process

### Step 0: Refresh analytics snapshot

```bash
node scripts/analytics-snapshot.js
```

Fetches live data from PostHog (pageviews, top pages) and GSC (queries, positions, CTR). Skips any source whose API key is missing. Only refreshes if data is older than `stale_hours` (configured in `.pseo/analytics-config.json`). Use `--force` to always refresh.

Output: `.pseo/analytics-snapshot.json` — read this in Steps 2-6 to cross-reference content health with real traffic and ranking data.

### Step 1: Generate fresh keyword map

```bash
node scripts/keyword-map.js
```

### Step 2: Read keyword-map.json

Parse `public/assets/keyword-map.json` for cannibalization data.

### Step 3: Scan all content files

Read every JSON in `content/` and run checks 2-6 above.

### Step 4: Optional — Query Ahrefs MCP

If the Ahrefs MCP is available, run check 7.

### Step 5: Generate report

## Examples

### Example 1: "Run an audit"

1. Generate keyword map → 7 conflicts found
2. Scan 29 content files:
   - 4 pages have 0-1 interlinks
   - 2 pages have descriptions > 155 chars
   - All pages pass external reference check
3. Report with prioritized fixes
4. Direct user to `/seo-dashboard` for visual keyword tree

### Example 2: "Check cannibalization before I add more content to a vertical"

1. Focus on target vertical files only
2. Read keyword map → filter for conflicts in that vertical
3. Report: shared keywords between overlapping pages
4. Recommend: remove from one page, keep only in the most relevant page
5. User runs `improve-content` to fix before expanding

### Example 3: "Full audit with Ahrefs data"

1. Run standard audit (steps 1-6)
2. Query Ahrefs MCP for live rankings
3. Cross-reference: identify pages ranking #4-10 that can be pushed to top 3
4. Report includes both content-level and ranking-level fixes
