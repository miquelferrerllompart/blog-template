# Research Brief — Page Diagnostic Skill

## Overview

This skill generates a **research brief** for a specific page by reading all local data files. It presents the brief to the user, who can add their own context or goals, and then applies improvements based on the combined information.

**No API keys needed.** All data comes from committed files that the developer refreshes weekly.

### Relationship With Other Skills

| Skill | When to use | Handoff |
|-------|-------------|---------|
| `research-brief` (this) | Before improving a specific page | Presents brief → user adds context → apply edits directly |
| `improve-content` | Deep SEO-driven improvements with live Ahrefs data | Needs API keys — for the developer role |
| `content-audit` | Full blog-wide health check | Broader scope — audits all pages at once |

## When This Skill Triggers

- User says: "improve", "optimize", "work on", "fix" + a page slug or description
- User says: "research brief", "brief for", "analyze", "diagnostic" + a page reference
- User names a specific slug: "best-rewards-fashion-apparel"
- User describes a page: "the fashion rewards page", "the beauty loyalty ideas post"
- User says: "what should I improve?"

## Prerequisites

These files must exist (generated by the developer weekly):

| File | Required? | Purpose |
|------|-----------|---------|
| `content/{type}/{slug}.json` | Yes | The page content itself |
| `.pseo/analytics-snapshot.json` | Recommended | PostHog traffic + GSC rankings |
| `public/assets/keyword-map.json` | Recommended | Cannibalization data |
| `.pseo/keyword-metrics-cache.json` | Recommended | Ahrefs keyword metrics |

If analytics or keyword data is missing, note it in the brief and proceed with what's available.

---

## Step 1: Identify the Page

Determine which page the user wants to work on:

1. **By slug:** User gives exact slug → find JSON file in `content/`
2. **By description:** User describes the page → search content files for matching title/industry/type
3. **By URL:** User gives a URL → extract slug from path
4. **Unclear:** Ask the user which page they mean

Read the content JSON file completely.

## Step 2: Gather Local Data

Read these files (in parallel if possible):

1. **Content JSON** — the page itself (`content/{type}/{slug}.json`)
2. **Analytics snapshot** — `.pseo/analytics-snapshot.json`
3. **Keyword map** — `public/assets/keyword-map.json`
4. **Keyword cache** — `.pseo/keyword-metrics-cache.json`

## Step 3: Generate the Brief

Present the brief to the user in this exact format:

---

### Brief format

```markdown
# Research Brief: {page title}

## Page Overview

| Field | Value |
|-------|-------|
| **File** | `{relative path}` |
| **URL** | https://{your-domain}{page_url} |
| **Type** | {content_type} ({subtype}) |
| **Industry** | {industry_display} |
| **Title** | {seo.title} |
| **Description** | {seo.description} ({char count} chars) |

## Current Keywords

| Keyword | Volume | KD | CPC |
|---------|--------|-----|-----|
| {keyword} | {vol or —} | {kd or —} | {cpc in USD or —} |
(for each keyword in seo.keywords, look up metrics in keyword-metrics-cache.json)

## Traffic & Rankings

### PostHog ({period})
- **Pageviews:** {views or "0 — not in top pages"}

### Google Search Console ({period})
- **Clicks:** {clicks or —}
- **Impressions:** {impressions or —}
- **CTR:** {ctr or —}
- **Avg Position:** {position or —}

**Top queries:**
| Query | Clicks | Impressions | CTR | Position |
(if GSC data exists for this page's URL)

*If no GSC data: "No GSC data — may not be indexed yet."*

## Keyword Cannibalization

(Read keyword-map.json → find conflicts involving this page's slug)

| Keyword | Risk | Competing Pages |
|---------|------|-----------------|
| {kw} | {HIGH/MEDIUM} | {other slugs} |

*If no conflicts: "No conflicts found."*

## Content Health

| Metric | Current | Target | Status |
|--------|---------|--------|--------|
| Interlinks | {count <a href in content} | 2-4 | {OK/LOW/HIGH} |
| External refs | {count external_references} | 3-5 | {OK/LOW/HIGH} |
| Pro tips | {count} | 5 | {OK/LOW/HIGH} |
| Common mistakes | {count} | 3 | {OK/LOW/HIGH} |
| FAQ items | {count} | 5-6 | {OK/LOW/HIGH} |
| Sections | {count} | 3-5 | {OK/LOW/HIGH} |
| SEO description | {chars} chars | ≤155 | {OK/TOO LONG} |
| Keywords count | {count} | 8-12 | {OK/LOW/HIGH} |

### External References
| # | Source | URL |
(list each external reference)

## Related Pages (same industry)

(from keyword-map.json tree, list pages in the same industry with links)

**{type}:**
- `{slug}` — {title}

## Keyword Opportunities

Keywords in the cache that are relevant to this industry but NOT used by this page:

| Keyword | Volume | KD | Traffic Potential |
(filter keyword-metrics-cache.json for industry-related terms with volume > 0, max 15, sorted by volume desc)

## Suggested Actions

(Prioritized list based on the data above)
- **{action}:** {description}
```

---

### How to count interlinks

Count occurrences of `<a href` (or `<a href=\"` in raw JSON) within the `content` object of the JSON. These are contextual internal links within body text fields.

### How to find keyword metrics

For each keyword in `seo.keywords`, look it up (lowercase, trimmed) in `.pseo/keyword-metrics-cache.json`. CPC values in the cache are in USD cents — divide by 100 to display in USD.

### How to find analytics data

In `.pseo/analytics-snapshot.json`:
- **PostHog:** Search `posthog.top_pages` for entries whose `url` contains the page's URL path
- **GSC:** Search `gsc.top_queries` and `gsc.top_pages` for entries matching the page's URL

### How to find cannibalization

In `public/assets/keyword-map.json`:
- Read `conflicts` array
- Filter for conflicts where any entry in `pages` has matching `slug`

### How to find keyword opportunities

In `.pseo/keyword-metrics-cache.json`:
- Get the page's industry terms (split industry slug by `-`, filter words > 2 chars)
- Also include general terms relevant to your niche (configured in taxonomy files)
- Filter cache entries that match these terms AND have `volume > 0` AND are NOT in the page's current `seo.keywords`
- Sort by volume descending, take top 15

### How to determine suggested actions

Generate actions based on these rules:

| Condition | Action |
|-----------|--------|
| Interlinks < 2 | "Add interlinks: add contextual `<a href>` links to related pages listed above" |
| Interlinks > 4 | "Reduce interlinks: remove the least relevant ones" |
| External refs < 3 | "Add external references: need at least 3 (Tier 1: Forbes, HBR, major industry authorities)" |
| SEO description > 155 chars | "Shorten SEO description: currently {N} chars (max 155)" |
| Keywords < 8 | "Add keywords: check Keyword Opportunities section" |
| Keywords > 12 | "Trim keywords: remove least relevant ones" |
| Pro tips ≠ 5 | "Fix pro tips count: currently {N} (target: 5)" |
| Common mistakes ≠ 3 | "Fix common mistakes count: currently {N} (target: 3)" |
| FAQ < 5 | "Add FAQ items: currently {N} (target: 5-6)" |
| 0 PostHog views | "Zero traffic: check if indexed, verify keywords have search volume" |
| GSC position 4-10 | "Quick win: small content improvements can push to top 3" |
| GSC high impressions, low CTR | "Low CTR: rewrite title and description to match top queries" |
| All external refs from same source | "Diversify references: all from {source}, add other Tier 1 sources" |

## Step 4: Wait for User Input

After presenting the brief, ask:

> **What would you like to focus on?** You can:
> - Pick specific actions from the list above
> - Add your own goals (e.g., "target the keyword X", "make it more specific to fast fashion")
> - Say "fix everything" and I'll work through all suggested actions

**Do NOT start editing until the user responds.** The brief is a conversation starter, not an automatic fix.

## Step 5: Apply Improvements

Once the user confirms or adds context:

1. Edit the JSON file following all schema rules (see `.pseo/schemas/`)
2. Only change what the user approved or requested
3. Maintain all existing schema constraints
4. Run `scripts/validate.sh` to verify
5. Run `scripts/build.sh` to rebuild
6. Present a summary of what changed

### Summary format

```
## Changes Applied: {slug}

| What | Before | After |
|------|--------|-------|
| SEO description | {old} ({chars}) | {new} ({chars}) |
| Keywords | {removed: [...], added: [...]} | |
| Interlinks | {N} → {N} | |
| {other changes} | | |

Next: run `bash scripts/validate.sh && bash scripts/build.sh` to rebuild.
```

## Examples

### Example 1: "I want to improve a resource page"

1. Read the target content JSON file
2. Read analytics, keyword map, keyword cache
3. Present brief showing: SEO description too long (157 chars), 0 FAQ items, zero traffic, keyword opportunities available
4. **Wait for user response**
5. User says: "Fix the description and add FAQs"
6. Shorten description to ≤155 chars, add 5 FAQ items relevant to the topic
7. Validate and rebuild

### Example 2: "What should I work on today?"

1. Read `.pseo/analytics-snapshot.json` for pages with issues
2. Read `public/assets/keyword-map.json` for cannibalization conflicts
3. Present a quick summary: "3 pages with zero traffic, 2 high-risk cannibalization conflicts, 5 pages with low interlinks"
4. User picks one → generate full brief for that page

### Example 3: "Improve content in a specific vertical — focus on a subtopic"

1. Find the target content JSON file
2. Generate full brief
3. User adds: "I want to target '{keyword}' as the primary keyword"
4. Apply changes: update seo.keywords, adjust title if needed, add content sections targeting the new keyword
